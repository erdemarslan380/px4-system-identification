PX4 / Gazebo System Identification and Digital Twin Workflow
===========================================================

1. Objective
------------
This repository implements a PX4-based system-identification workflow for multirotors. The practical goal is simple:
- execute a small set of physically meaningful OFFBOARD maneuvers,
- log the signals needed to estimate mass, inertia, drag, and motor-model terms,
- write the identified parameters back into a Gazebo SDF model,
- then check whether the resulting digital twin reproduces flight behavior on unseen trajectories.

The long-term target is real-flight validation. The immediate target in this repository is stricter:
- the method should recover the hidden x500 Gazebo SDF parameters almost exactly in truth-assisted SITL,
- because without that upper-bound result there is no reason to trust the method on real hardware.

2. Current headline result
--------------------------
The current truth-assisted SITL upper bound now satisfies that requirement.

Frozen upper-bound candidate:
- `examples/paper_assets/candidates/x500_truth_assisted_sitl_v1/`

Comparable x500 SDF terms now match to numerical precision for:
- mass
- `Ixx`, `Iyy`, `Izz`
- `time_constant_up`, `time_constant_down`
- `motor_constant`, `moment_constant`
- `rotor_drag_coefficient`, `rolling_moment_coefficient`
- `rotor_velocity_slowdown_sim`

Current upper-bound blended twin score:
- `99.99999999996612 / 100`

This result means:
- the estimator and SDF comparison pipeline are now behaving as they should in truth-assisted SITL,
- remaining future mismatch in real-flight studies should be treated as an observability, logging, or sortie-design problem,
- not as a core estimator failure.

3. System architecture
----------------------
The stack has three layers.

3.1 PX4 runtime layer
- `overlay/src/modules/custom_pos_control/*`
  - minimal OFFBOARD forwarder
  - supports only `px4_default` and `sysid`
- `overlay/src/modules/trajectory_reader/*`
  - identification profile generator
  - tracking and identification logging
- `overlay/src/modules/simulation/gz_plugins/system_identification_logger/*`
  - Gazebo truth logger for simulator-only validation

3.2 Sortie execution layer
- PX4 SITL or real PX4 firmware runs the same identification motions.
- In SITL the vehicle is the Gazebo `x500`.
- In real flight the operator performs the same sortie structure, but only PX4-side logs are available.

3.3 Offline estimation layer
- `experimental_validation/identification.py`
  parses logs and estimates physical parameters
- `experimental_validation/estimators.py`
  reusable regressors and closed-form estimators
- `experimental_validation/compare_with_sdf.py`
  compares identified parameters against the x500 SDF
- `experimental_validation/paper_artifacts.py`
  generates validation figures for the paper

4. Identification maneuver library
----------------------------------
The digital twin is not estimated from one large trajectory. It is identified from several short families, each designed to make one parameter group observable.

4.1 Hover thrust
- profile: `hover_thrust`
- duration: about `26 s`
- purpose:
  - hover-force balance
  - thrust scaling

4.2 Mass vertical
- profile: `mass_vertical`
- duration: about `36 s`
- purpose:
  - stronger vertical excitation for mass observability

4.3 Roll sweep
- profile: `roll_sweep`
- duration: about `28 s`
- purpose:
  - roll-axis inertia excitation

4.4 Pitch sweep
- profile: `pitch_sweep`
- duration: about `28 s`
- purpose:
  - pitch-axis inertia excitation

4.5 Yaw sweep
- profile: `yaw_sweep`
- duration: about `24 s`
- purpose:
  - yaw-axis inertia and yaw torque excitation

4.6 Drag X / Y / Z
- profiles: `drag_x`, `drag_y`, `drag_z`
- duration: about `30 s` each
- purpose:
  - aerodynamic drag identification along each body axis

4.7 Motor step
- profile: `motor_step`
- duration: about `24 s`
- purpose:
  - `time_constant_up`
  - `time_constant_down`
  - `motor_constant`
  - `moment_constant`
  - `rotor_drag_coefficient`
  - `rolling_moment_coefficient`
  - `rotor_velocity_slowdown_sim`
  - maximum rotor speed summary

5. Logged signals
-----------------
Two logs are used.

5.1 PX4 identification CSV
Produced by `trajectory_reader`.
This is the key log for real flight.
Main content:
- references:
  - position, velocity, acceleration, yaw, yaw-rate
- measured states:
  - position, velocity, acceleration
  - roll, pitch, yaw
  - body rates
- controller / actuator side:
  - thrust setpoint
  - attitude setpoint
  - body-rate setpoint
  - motor outputs
  - ESC rpm, voltage, current
- metadata:
  - active profile
  - controller label
  - hover thrust summary

5.2 Gazebo truth CSV
Produced by `SystemIdentificationLoggerPlugin`.
This is the simulator-only upper-bound validation log.
Main content:
- total propulsive force
- body torques
- angular accelerations
- actual rotor speeds
- per-rotor thrust summaries
- motor plugin terms used by Gazebo

6. Identification mathematics
-----------------------------
6.1 Mass and thrust scale
Mass is estimated from the vertical force balance. The current implementation is tilt-aware: it uses the effective vertical thrust component rather than assuming the vehicle remains perfectly level.

6.2 Inertia
The inertia block compares two models and keeps the better physically valid fit:
- simple axis-wise fit
- coupling-aware fit with nuisance cross terms

Conceptually:
- `tau_x ~= Ixx * p_dot + kx * q * r`
- `tau_y ~= Iyy * q_dot + ky * p * r`
- `tau_z ~= Izz * r_dot + kz * p * q`

This matters because roll identification can be biased if the maneuver is not perfectly decoupled.

6.3 Drag
Quadratic drag is estimated per axis from velocity-dependent force terms.

6.4 Motor model
Motor time constants and motor coefficients are estimated from `motor_step` sorties.
A critical implementation detail is run-boundary segmentation: multiple sorties must not be treated as one continuous time axis.

7. Why the current upper-bound finally became exact
---------------------------------------------------
The last non-negligible gap in truth-assisted SITL was not a physics problem anymore. It was a logging precision problem.

Root cause:
- Gazebo truth terms such as `motor_constant` were written with fixed decimal formatting after the first row,
- tiny values were rounded,
- the estimator therefore saw slightly corrupted truth coefficients.

Fix:
- the Gazebo truth logger now writes truth fields in high-precision scientific notation.

Effect:
- `motor_constant` and the remaining comparable motor-model terms now recover exactly in the truth-assisted SITL upper bound,
- the blended score moved from about `90.28` to essentially `100`.

8. Two-stage validation flow for the paper
------------------------------------------
The paper should present the digital-twin validation in two stages.

8.1 Stage 1: flight-to-twin trajectory comparison
This stage answers:
- once identification is done, does the twin reproduce unseen mission trajectories?

The repository already generates placeholder figures for five trajectories:
- `hairpin`
- `lemniscate`
- `circle`
- `time_optimal_30s`
- `minimum_snap_50s`

Current figures use synthetic noisy stand-ins for real-flight logs. Later you replace only the CSV inputs and regenerate the same plots.

Stage-1 figures include:
- 3D overlays
- error-norm traces
- a summary score chart

9. Stage 2: SITL robustness and stress tests
--------------------------------------------
This stage answers:
- if the plant changes, how quickly does the identified twin degrade?

The twin is kept fixed while the reference plant is perturbed.
The current stress-test families are:
- payload mass vs payload z-offset
- payload mass vs payload x-offset
- payload mass vs payload y-offset
- payload mass vs arm-length scale
- motor-constant scale vs motor-time scale

The current stress-test visuals are intentionally not pretending to be real-flight truth.
Their role is different:
- they visualize robustness,
- they show where the twin is sensitive,
- they give the paper a meaningful model-variation section before real-flight data arrive.

10. Real-flight sortie plan
---------------------------
The real-flight protocol should stay short and battery-aware. A practical decomposition is:

Sortie 1: hover and mass
- manual takeoff
- climb to about `3 m` AGL
- baseline PX4 PID active
- enter OFFBOARD
- run `hover_thrust`
- run `mass_vertical`
- return to hover
- exit OFFBOARD and land

Sortie 2: roll and pitch inertia
- manual takeoff to about `3 m`
- baseline PX4 PID active
- enter OFFBOARD
- run `roll_sweep`
- hover recovery
- run `pitch_sweep`
- hover recovery
- land

Sortie 3: yaw and motor dynamics
- manual takeoff to about `3 m`
- enter OFFBOARD
- run `yaw_sweep`
- hover recovery
- run `motor_step`
- hover recovery
- land

Sortie 4: drag
- manual takeoff to about `3 m`
- enter OFFBOARD
- run `drag_x`
- hover recovery
- run `drag_y`
- hover recovery
- run `drag_z`
- hover recovery
- land

Validation sorties after identification:
- one or two additional sorties with the five unseen validation trajectories

11. Manual SITL execution without AI
------------------------------------
To watch the motions in Gazebo while the repository is being used manually:
1. sync the overlay into PX4
2. build `make px4_sitl gz_x500`
3. run with GUI enabled:
   - `unset HEADLESS`
   - `make px4_sitl gz_x500`
4. if needed, open the GUI with:
   - `gz sim -g`
5. in the PX4 shell:
   - `custom_pos_control start`
   - `trajectory_reader start`
   - `custom_pos_control enable`
   - `custom_pos_control set sysid`
   - `trajectory_reader set_mode identification`
   - `trajectory_reader set_ident_profile <profile_name>`

12. Honest current claim
------------------------
An honest publication-ready statement today is:
- the truth-assisted SITL upper bound has been solved to numerical precision,
- the paper-figure pipeline is complete,
- the stress-test and trajectory-comparison visuals are available,
- the real-flight logging path is defined,
- but the true real-flight digital-twin claim still requires replacing the synthetic placeholder trajectory logs with actual outdoor flight logs.
