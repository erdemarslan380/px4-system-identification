PX4 system identification and digital-twin methodology
======================================================

Purpose
-------
This text explains exactly how the repository identifies a multicopter model from PX4 logs, turns the result into a Gazebo SDF candidate, and validates that candidate as a digital twin. It is written as source material for a later paper, thesis chapter, or prompt-based writing workflow.

Repository scope
----------------
This repository contains only the system-identification workflow. It does not contain the separate optimization stack. The pipeline is intentionally split into two parts:
- `overlay/`: PX4 and Gazebo code that creates the maneuvers and records the logs.
- `experimental_validation/`: the offline estimators, SDF comparison scripts, scoring logic, and figure generators.

Core runtime components
-----------------------
1. `overlay/src/modules/custom_pos_control/custom_pos_control.cpp`
   - the minimal control-side entry point used in this repository.
   - for system identification, the important setting is `sysid` mode.
2. `overlay/src/modules/trajectory_reader/trajectory_reader.cpp`
   - publishes the reference setpoints for either trajectory playback or system-identification maneuvers.
   - writes PX4-side tracking logs and identification logs.
3. `overlay/src/modules/simulation/gz_plugins/system_identification_logger/SystemIdentificationLogger.cpp`
   - logs Gazebo truth values during SITL.
   - provides the simulator-side upper-bound reference.

Core offline components
-----------------------
1. `experimental_validation/cli.py`
   - estimates parameters from one maneuver log pair.
2. `experimental_validation/identification.py`
   - orchestrates full identification runs and prepares merged data tables.
3. `experimental_validation/estimators.py`
   - contains the actual mass, inertia, drag, and motor-model estimators.
4. `experimental_validation/compare_with_sdf.py`
   - compares identified parameters with the reference x500 SDF.
5. `experimental_validation/twin_metrics.py`
   - computes the blended digital-twin score and the family-level scores.
6. `experimental_validation/paper_artifacts.py`
   - creates trajectory overlays, robustness plots, error bars, and summary files.

Method overview
---------------
The identification method is built around a family of short, targeted maneuvers rather than a single monolithic experiment. Each maneuver is designed to isolate one parameter family as much as possible.

The nine built-in maneuvers are:
- `hover_thrust`
- `mass_vertical`
- `roll_sweep`
- `pitch_sweep`
- `yaw_sweep`
- `drag_x`
- `drag_y`
- `drag_z`
- `motor_step`

Each maneuver is triggered directly from the PX4 shell through `trajectory_reader set_ident_profile <profile>`.

Exact SITL execution flow
-------------------------
1. Sync the overlay into an upstream PX4 tree:
   - `~/px4-system-identification/sync_into_px4_workspace.sh ~/PX4-Autopilot`
2. Build and start Gazebo SITL:
   - `cd ~/PX4-Autopilot`
   - `unset HEADLESS`
   - `make px4_sitl gz_x500`
3. In the PX4 shell:
   - `custom_pos_control start`
   - `trajectory_reader start`
   - `custom_pos_control enable`
   - `custom_pos_control set sysid`
   - `trajectory_reader set_mode identification`
   - `trajectory_reader set_ident_profile hover_thrust`
4. From QGroundControl:
   - arm
   - take off manually to about `3 m`
   - stabilize hover
   - switch to `OFFBOARD`
5. Run one maneuver at a time and wait for completion before sending the next one.

PX4 console behavior during a maneuver
--------------------------------------
The current implementation prints the operational state explicitly:
- `Identification profile set to ...`
- `Purpose: ...`
- `Estimated duration: ...`
- `Identification maneuver started: ...`
- `Identification log started: ...`
- `Identification maneuver completed: ...`
- `Identification log completed: ...`
- `Tracking log completed: ...`

This means the operator does not need to guess when a maneuver ends. The end-of-maneuver condition is now visible from the PX4 console itself.

Maneuver definitions
--------------------
1. `hover_thrust`
   - purpose: excite the hover-thrust estimator around a steady hover point.
   - duration: about `26 s`.
   - output family: hover thrust tracking and part of thrust scaling.
   - after completion: the vehicle holds the last reference.

2. `mass_vertical`
   - purpose: excite the vertical dynamics with a multi-frequency vertical command.
   - duration: about `36 s`.
   - output family: total mass and thrust scale.
   - after completion: the vehicle holds the last reference.

3. `roll_sweep`
   - purpose: excite roll-axis inertia and cross-axis coupling through lateral motion.
   - duration: about `28 s`.
   - output family: primarily `Ixx` and roll coupling terms.
   - after completion: the vehicle holds the last reference.

4. `pitch_sweep`
   - purpose: excite pitch-axis inertia and coupling.
   - duration: about `28 s`.
   - output family: primarily `Iyy` and pitch coupling terms.
   - after completion: the vehicle holds the last reference.

5. `yaw_sweep`
   - purpose: excite yaw inertia and yaw moment balance.
   - duration: about `24 s`.
   - output family: primarily `Izz` and yaw-related moment terms.
   - after completion: the vehicle holds the last reference.

6. `drag_x`
   - purpose: excite the X-axis translational drag.
   - duration: about `30 s`.
   - output family: forward drag-related terms.
   - after completion: the vehicle holds the last reference.

7. `drag_y`
   - purpose: excite the Y-axis translational drag.
   - duration: about `30 s`.
   - output family: lateral drag-related terms.
   - after completion: the vehicle holds the last reference.

8. `drag_z`
   - purpose: excite the Z-axis drag and vertical damping.
   - duration: about `30 s`.
   - output family: vertical drag-related terms.
   - after completion: the vehicle holds the last reference.

9. `motor_step`
   - purpose: excite actuator dynamics through a step-like thrust profile.
   - duration: about `24 s`.
   - output family: motor time constants and motor dynamic terms.
   - after completion: the vehicle holds the last reference.

Recorded quantities
-------------------
The PX4 identification CSV includes, at minimum:
- reference position, velocity, acceleration, yaw, and yaw rate,
- vehicle position, velocity, acceleration,
- attitude and body rates,
- attitude setpoint and rate setpoint,
- actuator outputs,
- ESC rpm, voltage, and current if available,
- hover thrust estimate,
- controller label.

The Gazebo truth CSV includes simulator-side quantities such as:
- body accelerations,
- rotor speeds,
- thrust-related truth terms,
- vehicle truth state.

Exact SITL log directories
--------------------------
- PX4 identification logs:
  - `~/PX4-Autopilot/build/px4_sitl_default/rootfs/identification_logs/`
- PX4 tracking logs:
  - `~/PX4-Autopilot/build/px4_sitl_default/rootfs/tracking_logs/`
- Gazebo truth logs:
  - `~/PX4-Autopilot/build/px4_sitl_default/rootfs/sysid_truth_logs/`

On hardware, the corresponding PX4 storage directories are typically:
- `/fs/microsd/identification_logs/`
- `/fs/microsd/tracking_logs/`

Offline estimation flow
-----------------------
The offline estimation is performed in three conceptual layers.

Layer 1: one-log estimation
- `experimental_validation/cli.py` reads one PX4 identification CSV and, optionally, one Gazebo truth CSV.
- It writes:
  - `identified_parameters.json`
  - `candidate_inertial.sdf.xml`
  - `candidate_vehicle_params.yaml`

Layer 2: full maneuver-family fusion
- `experimental_validation/compare_with_sdf.py` combines the maneuver family:
  - `hover_thrust`
  - `mass_vertical`
  - `roll_sweep`
  - `pitch_sweep`
  - `yaw_sweep`
  - `drag_x`
  - `drag_y`
  - `drag_z`
  - `motor_step`
- It then compares the candidate with the x500 SDF and writes:
  - `identified_parameters.json`
  - `identified_parameters_by_mode.json`
  - `sdf_reference.json`
  - `sdf_comparison.json`
  - `sdf_comparison_by_mode.json`
  - `used_identification_logs.json`
  - `candidate_x500_base.sdf`

Layer 3: figure generation and statistical validation
- `experimental_validation/paper_artifacts.py` generates:
  - five trajectory overlay plots,
  - stress-test surface plots,
  - stress-test line plots,
  - parameter error bars,
  - family score bars,
  - trajectory summary score plots,
  - `paper_validation_summary.json`.

Validation modes
----------------
Three validation modes are supported:
- `px4_only`
- `telemetry_augmented`
- `truth_assisted`

Interpretation:
- `px4_only` is the most realistic mode for real-flight transfer.
- `telemetry_augmented` adds measurements that are still feasible to log outside the simulator.
- `truth_assisted` is the simulator-side upper bound. If this mode cannot recover the hidden x500 parameters, the method is not ready.

Current strongest claim
-----------------------
At the current checkpoint, the strongest validated claim is:
- truth-assisted SITL can recover the comparable x500 SDF terms to numerical precision,
- the figure-generation pipeline is complete,
- the repository can now show the complete operator flow from maneuver execution to SDF comparison and paper figures.

This is the correct upper-bound milestone. It does not automatically imply that real-flight digital-twin equivalence has already been proven.

Five validation trajectories
----------------------------
The repository uses five validation trajectories:
- `hairpin`
- `lemniscate`
- `circle`
- `time_optimal_30s`
- `minimum_snap_50s`

These are intentionally defined so that the logged mission segment:
- starts from the same XY point,
- starts from the same altitude (`3 m`, meaning `z = -3` in NED),
- returns to the same logged start pose.

Operational rule for both SITL and real flight:
1. take off and stabilize at about `3 m`,
2. move to the common start pose,
3. begin logging only after the vehicle reaches that pose,
4. execute the trajectory,
5. return to hover.

This design makes trajectory overlays meaningful because the logged segments share the same spatial reference.

Two-stage paper result flow
---------------------------
Stage 1: real-flight versus digital-twin overlays
- identify the model from the four identification sorties,
- write the estimated parameters into a Gazebo SDF candidate,
- run the five validation trajectories in simulation,
- collect the corresponding real-flight trajectory logs,
- overlay real flight and digital twin on the same axes,
- report RMSE, delay, and blended twin score.

Stage 2: SITL statistical stress tests
- keep the identified twin fixed,
- perturb the reference plant across multiple families,
- generate surfaces and slices for:
  - payload mass versus payload Z offset,
  - payload mass versus payload X offset,
  - payload mass versus payload Y offset,
  - payload mass versus arm-length scale,
  - payload mass versus motor-dynamics scale,
- use those plots to show robustness and failure sensitivity.

Important interpretation of Stage 2
-----------------------------------
These surfaces are not meant to claim real-flight equivalence by themselves. They are robustness plots. Their role is to show how the fixed identified twin behaves when the plant is perturbed.

Real-flight sortie plan
-----------------------
Recommended calm-day sequence:
- Sortie 1: `hover_thrust`, `mass_vertical`
- Sortie 2: `roll_sweep`, `pitch_sweep`
- Sortie 3: `yaw_sweep`, `motor_step`
- Sortie 4: `drag_x`, `drag_y`, `drag_z`
- Sortie 5 and later: the five validation trajectories

Every sortie uses the same operational rule:
- start helper modules,
- take off manually,
- stabilize at about `3 m`,
- enter OFFBOARD,
- run one profile,
- wait for the completion messages,
- recover to hover,
- continue only if the safety and battery margin remain good.

SDF update workflow
-------------------
After estimation, the inertial snippet written by the tool is stored in `candidate_inertial.sdf.xml`. The recommended practical workflow is:
1. duplicate the stock x500 Gazebo model into a new model folder such as `~/PX4-Autopilot/Tools/simulation/gz/models/x500_identified/`,
2. replace the relevant inertial and motor parameters with the identified values,
3. leave the stock x500 model untouched for comparison,
4. rerun the five validation trajectories with the identified model,
5. generate the overlay figures again.

Real-flight calibration restore after firmware update
-----------------------------------------------------
This repository also supports restoring calibration-related parameters after a firmware update.

Place the QGroundControl export at:
- `~/px4-system-identification/experimental_validation/qgc/current_vehicle.params`

Then run:
- `python3 experimental_validation/calibration_restore.py --input ~/px4-system-identification/experimental_validation/qgc/current_vehicle.params --out-dir ~/px4-system-identification/experimental_validation/qgc/restore`
